## **Входные данные (что получает инструмент)**

Пользователь должен иметь возможность задать **что тестировать**, **как тестировать** и **в каких условиях**.

### **1\. Спецификация тестируемого API (обязательно)**

-  **OpenAPI / Swagger** (JSON/YAML) -- основной источник структуры эндпоинтов, методов, параметров, схем данных.

-  **Postman Collection** (v2) -- альтернативный формат.

-  **GraphQL Schema** (SDL или introspection JSON) -- если поддержка GraphQL.

-  **gRPC .proto файлы** (опционально).

-  **Ручное описание** -- через GUI или код (DSL).

### **2\. Конфигурация тестирования (что и как проверяем)**

-  **Профили нагрузки**:

   -  Виртуальные пользователи / потоки.

   -  RPS (запросов в секунду) -- фиксированный или рамп-ап.

   -  Длительность теста.

   -  Таймауты, задержки между запросами (think time).

-  **Сценарии безопасности**:

   -  Типы атак: SQLi, XSS, XXE, SSTI, LDAP injection.

   -  Словари для фаззинга (можно кастомизировать).

   -  Проверки авторизации: список пар ролей / пользователей, маппинг эндпоинтов на разрешенные роли.

-  **Проверки конфиденциальных данных**:

   -  Регулярные выражения для поиска паттернов (email, телефон, паспорт, кредитка).

   -  Словари ключей (например, `password`, `secret`, `token`).

-  **Политики HTTP/TLS**:

   -  Требуемые версии TLS (1.2, 1.3).

   -  Список разрешенных шифров.

   -  Обязательные заголовки безопасности (HSTS, CSP, X-Frame-Options).

-  **Условия успеха**:

   -  Пороговые значения (например, p95 latency ≤ 300ms, error rate ≤ 1%).

   -  Ожидаемые HTTP статусы.

### **3\. Тестовые данные**

-  **Наборы данных** (CSV, JSON, YAML) для параметризации запросов (например, тестовые пользователи, идентификаторы).

-  **Файлы** для загрузки (обычные и вредоносные, например, EICAR).

-  **Seed-данные** (скрипты для наполнения БД перед тестом) -- опционально.

### **4\. Конфигурация окружения**

-  **Базовый URL** (или несколько для разных сред: dev, staging, prod).

-  **Аутентификация**:

   -  Basic Auth (username/password).

   -  Bearer Token (статический или скрипт получения).

   -  OAuth2 client credentials / authorization code flow.

   -  API Key (header или query).

-  **Заголовки** (по умолчанию для всех запросов).

-  **Прокси** и **сертификаты** для работы с защищенными сетями.

-  **Переменные окружения** (подстановка в запросах).

## **Выходные данные (что генерирует инструмент)**

Результаты должны быть структурированными, наглядными и пригодными для автоматической обработки.

-  Формат отчетов (HTML, JSON, JUnit XML, CSV, PDF).

-  Куда сохранять результаты (локальная папка, S3, артефакты CI).

-  Уведомления: email, Slack, webhook, создание issue в Jira/GitHub.

### **1\. Основной отчет (сводка)**

-  **Общая информация**: дата теста, версия инструмента, цель теста (URL), конфигурация.

-  **Общий вердикт**: PRODUCTION_READY / HAS_WARNINGS / FAILED.

-  **Количество проверок**: всего, пройдено, провалено, с ошибками.

-  **Время выполнения теста**.

### **2\. Результаты производительности (нагрузочное тестирование)**

-  **Метрики по каждому эндпоинту**:

   -  Запросов всего, успешных (2xx, 3xx), ошибок (4xx, 5xx).

   -  Latency: min, max, mean, p50, p95, p99, стандартное отклонение.

   -  Throughput (RPS).

-  **Временной ряд** (можно в JSON или CSV для построения графиков):

   -  Таймстемп, количество запросов, средняя задержка, количество ошибок.

-  **Графики** (если инструмент генерирует визуализации):

   -  График задержек по времени.

   -  Гистограмма распределения времени ответа.

   -  График RPS и ошибок.

-  **Топ медленных эндпоинтов** (список).

### **3\. Результаты проверок безопасности**

-  **Обнаруженные уязвимости** (структурировано):

   -  Название (SQL Injection, IDOR, XSS и т.д.).

   -  Severity: Critical, High, Medium, Low.

   -  Эндпоинт и метод.

   -  Параметр, в котором найдено.

   -  Вредоносный запрос (payload).

   -  Ответ сервера (сниппет, где видно уязвимость).

   -  Доказательство (proof) -- например, разница во времени или вывод данных.

-  **Проверенные заголовки безопасности**: какие присутствуют, какие нет, их значения.

-  **Результаты TLS сканирования**: версия, шифры, уязвимости (если есть).

-  **Результаты IDOR тестирования**: если удалось получить чужие данные (с указанием пользователя и ресурса).

### **4\. Результаты анализа конфиденциальных данных**

-  **Совпадения по паттернам**:

   -  Тип данных (email, credit card, passport).

   -  Эндпоинт, где найден.

   -  Пример найденной строки (с маскировкой по желанию).

   -  Поле ответа (JSON path или тело).

-  **Предупреждения** о передаче данных в URL (GET запросы с конфиденциальными параметрами).

### **5\. Observability / трассировка (уникальный выход)**

-  **Отчет по трассировкам** для подозрительно медленных запросов:

   -  Trace ID, длительность, breakdown по спанам.

   -  Где больше всего времени (БД, внешний сервис, очередь).

   -  Ошибки в спанах (например, исключение в сервисе).

-  **Проверка корректности propagation**:

   -  Какие запросы не имеют входящего trace header.

   -  Где потерян контекст.

### **6\. Логи и сырые данные**

-  **Полный лог всех запросов и ответов** (включая тело) -- по желанию (может быть огромным).

-  **Артефакты** для воспроизведения:

   -  Скрипт (cURL, Python) для каждого нашедшегося бага.

   -  Docker-команда для запуска инструмента с теми же параметрами.

-  **Файл результатов** в JSON -- для интеграции с другими системами.

### **7\. Интеллектуальные рекомендации (AI-enhanced)**

-  **Анализ причин падения**:

   *"Высокая задержка на /api/report вызвана синхронным вызовом внешнего сервиса (см. trace xxx)"*.

-  **Приоритет исправлений** (ранжирование найденных проблем).

-  **Сравнение с предыдущим тестом** (регрессии производительности).
