**REST API** (Representational State Transfer Application Programming Interface) является доминирующим архитектурным стилем для взаимодействия между современными веб‑приложениями, микросервисами и мобильными клиентами. API обеспечивают программный доступ к бизнес‑логике и данным, что делает их критически важным компонентом информационной инфраструктуры.

Вместе с ростом числа и сложности API увеличивается и **поверхность атаки**. Согласно отчёту Imperva «The State of API Security in 2024», API‑трафик составляет более 71% всего веб‑трафика, при этом:

-  в 2023 году 27% всех атак на API были направлены на **злоупотребление бизнес‑логикой** – на 10% больше, чем годом ранее;

-  19% атак совершено автоматизированными агентами (вредоносными ботами);

-  45,8% атак на учётные записи (Account Takeover) приходится на API‑эндпоинты.

В OWASP API Security Top 10 (2023) первое место занимает **Broken Object Level Authorization (BOLA)** – уязвимость, позволяющая злоумышленнику получить несанкционированный доступ к объектам других пользователей через манипуляцию идентификаторами. Кроме того, в перечень входят: неограниченное потребление ресурсов, избыточное раскрытие данных, массовое присвоение (mass assignment) и небезопасное использование API.

**Нефункциональное тестирование безопасности** включает проверки, которые не сводятся к проверке корректности работы по спецификации. Оно охватывает:

-  **нагрузочное тестирование** – оценку поведения API при высоком количестве одновременных запросов;

-  **фаззинг** – подачу случайных или специально сформированных вредоносных данных для выявления сбоев и уязвимостей (ГОСТ Р 56939‑2024, п. 3.22);

-  **анализ заголовков безопасности** – проверку наличия и корректности HTTP‑заголовков, влияющих на безопасность;

-  **проверку утечек данных** – обнаружение конфиденциальной информации (пароли, токены, персональные данные) в ответах API.

Несмотря на наличие отдельных инструментов для нагрузочного тестирования (JMeter, k6) и для безопасности (ZAP, Burp Suite), на рынке отсутствует **единое кроссплатформенное решение**, которое сочетало бы оба класса проверок, позволяло импортировать OpenAPI‑спецификацию и хранило бы историю тестов с возможностью сравнения. Это определяет актуальность разработки ProofAPI.
